# Security Policy

## Overview

Orchestra is designed as a **local-first** control plane. Security and privacy are core principles. This document outlines the security measures and best practices for running Orchestra.

## Server Authentication

When running in server mode (`AI_SYSTEM_SERVER_MODE=true`), the following security measures apply:

- **Bearer Token:** All requests must include an `Authorization: Bearer <TOKEN>` header. The token is defined via the `AI_SYSTEM_SERVER_TOKEN` environment variable.
- **No Default Token:** Orchestra will refuse to start in server mode if a token is not provided.
- **Security Recommendation:** Use a long, random string generated by a password manager.

## Local Filesystem Access

The server can perform write operations on your local filesystem.

- **Allowed Workdirs:** Use `AI_SYSTEM_ALLOWED_WORKDIRS` (comma-separated list of absolute paths) to restrict the server to specific directories.
- **Sandboxing:** Tool checks (lint, test) can be run inside **Docker** containers to isolate them from your host system. Set `tools.sandbox.mode: "docker"` in your `.ai-system.json`.
- **Atomic Writes:** Files are written using an atomic rename pattern to prevent corruption.

## Data Privacy

- **Local Providers:** By default, Orchestra uses local AI CLIs. Your code never leaves your machine unless you explicitly use a cloud-based provider like `9router` or `openai-compatible`.
- **Secret Redaction:** Orchestra automatically masks common API key formats in logs and webhooks.
- **Telemetry:** There is no built-in telemetry or "phone home" behavior. All execution data stays in `.ai-system-artifacts`.

## Network Security

- **Binding:** By default, the server binds to `0.0.0.0` inside Docker or `127.0.0.1` locally.
- **SSL/TLS:** The built-in server does not handle HTTPS. We recommend running Orchestra behind a reverse proxy (like Nginx, Caddy, or Cloudflare Tunnel) if you need to expose it over a network.

## Auditing

Every significant action (job creation, code write, approval) is recorded in `.ai-system-artifacts/audit-log.jsonl`. You can monitor this file or use the `/audit` API endpoint to track activity.
